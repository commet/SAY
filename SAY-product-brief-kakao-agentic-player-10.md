# 사이(SAY) 제품 기획서

작성일: 2026-07-04  
목적: 카카오 AGENTIC PLAYER 10 출품 및 구현을 위한 제품 기획 맥락 정리

---

## 1. 한 줄 정의

**사이**는 가족이 받은 병원, 관공서, 보험, 택배, 아파트 공지 같은 어려운 안내를 읽고, 가족끼리 서로 부담 없이 확인하고 완료까지 챙길 수 있게 돕는 카카오톡 AI다.

더 짧게:

> 가족이 받은 어려운 안내와 부탁을, 서로 바로 이해하고 함께 닫을 수 있는 행동 카드로 바꿔주는 카톡 AI.

---

## 2. 왜 이 제품인가

카카오톡은 가족의 생활 운영이 실제로 일어나는 곳이다. 부모님이 문자 캡처를 보내며 "이거 뭐야?", "사기 아니야?", "뭐 하라는 거야?"라고 묻고, 자녀나 다른 가족이 해석하고, 다시 설명하고, 알림을 챙기고, 처리 여부를 확인한다.

이 과정은 작지만 반복적이고, 놓치면 실제 손해가 생긴다.

- 병원 안내를 잘못 읽어 금식, 준비물, 도착 시간을 놓침
- 관공서/복지 안내의 신청 마감이나 필요 서류를 놓침
- 보험/카드/납부 문자를 이해하지 못해 불안해함
- 택배/결제/인증 문자가 스미싱인지 판단하기 어려움
- 가족 중 누가 확인하기로 했는지 흐려져서 결국 아무도 처리하지 않음

사이는 이 문제를 "요약"하지 않는다. 안내를 **행동 카드**로 바꾸고, 가족 사이의 빈칸을 보여주고, 확인과 완료까지 닫히게 한다.

---

## 3. 제품 태도

사이는 가족을 대신하지 않는다.  
사이는 감정적인 말을 흉내내지 않는다.  
사이는 누가 부족하다고 지적하지 않는다.  
사이는 조용히 사실, 빈칸, 다음 행동, 알림을 정리한다.

중요한 원칙:

> 가족을 대신하면 불쾌하고, 가족이 놓치지 않게 받쳐주면 좋다.

따라서 사이의 말투는 따뜻하되 사무적으로 정확해야 한다. 감정은 가족이 직접 덧붙일 수 있어야 한다.

좋은 예:

```text
안내문 기준으로는 금식 시간과 도착 시간이 확인됐어요.
복용 중인 약은 안내문에 없어 병원 확인이 필요해요.
```

피할 예:

```text
엄마를 위해 제가 꼼꼼히 챙겨드릴게요.
아빠, 걱정하지 마세요. 제가 다 알아서 할게요.
```

### 4.3 사이가 대회 의도와 맞는 이유

사이는 "AI 비서"나 "공지 요약기"가 아니다. 카카오톡에서 이미 반복되는 가족 간 해석, 확인, 전달, 알림 문제를 해결한다.

- **창의성**: 기관어/생활문서/가족 부탁을 가족 사이에서 실행 가능한 행동 카드로 바꾼다.
- **편의성**: 카톡에서 캡처나 문자를 전달하면 바로 결과를 받는다.
- **안정성**: 원문 기반 사실과 확인 필요 항목을 분리하고, 의료/법률/금융 판단은 하지 않는다.
- **카카오톡 적합성**: 가족 단톡방, 부모-자녀 대화, 캡처 전달이라는 기존 행동에 붙는다.

## 5. 핵심 문제

대부분의 안내문은 "읽힌 상태"에서 멈춘다.

사이는 안내문을 "닫힌 상태"로 만든다.

```text
읽음 -> 이해함 -> 빈칸 확인 -> 가족이 맡음 -> 알림 -> 완료
```

이 흐름이 제품의 중심이다.

---

## 6. 핵심 개념: 닫힌 안내

사이의 대표 개념은 **닫힌 안내**다.

안내문 하나가 들어오면 사이가 다음 상태를 만든다.

```text
상태: 확인 필요
마감: 2026-07-10 18:00
다음 확인: 오늘 20:00
남은 빈칸:
- 혈압약 복용 여부
- 누가 병원에 동행할지
```

각 항목은 상태를 가진다.

- 미확인
- 내가 확인할게요
- 가족에게 물어봤어요
- 확인 중
- 완료
- 보류
- 해당 없음

이 상태 모델이 있어야 서비스가 단순 요약을 넘어선다.

## 7. 대상 사용자

### 7.1 1차 사용자

- 부모님이 받은 문자나 공지를 대신 읽어주는 자녀
- 가족의 병원, 관공서, 보험, 납부, 택배 안내를 자주 챙기는 사람
- 가족 단톡방에서 "이거 누가 확인해?"를 자주 정리하는 사람

### 7.2 2차 사용자

- 부모님 또는 어르신 본인
- 배우자, 형제자매, 보호자
- 가족끼리 생활 일정을 공유하는 사람

### 7.3 중요한 관점

부모님은 수동적 도움 대상이 아니다. 부모님도 사이를 통해 자녀에게 명확하게 부탁할 수 있어야 한다.

예:

```text
민수에게 부탁하기:
"보험료 자동이체가 된 건지 확인해줄래?"
```

## 8. 사용자 여정

### 8.1 여정 A: 부모님이 받은 병원 안내를 가족이 함께 처리

1. 엄마가 병원 문자 캡처를 가족방이나 사이에 보낸다.
2. 사이가 OCR/비전으로 내용을 읽는다.
3. 사이가 문서 유형을 `병원/검진 안내`로 분류한다.
4. 사이가 행동 카드를 만든다.
5. 금식, 도착 시간, 준비물을 분리한다.
6. 복용 중인 약처럼 원문에 없는 정보는 `확인 필요`로 둔다.
7. 가족 중 누가 확인할지 자연스럽게 선택하게 한다.
8. 알림을 만든다.
9. 완료 여부를 나중에 묻는다.

출력 예:

```text
건강검진 안내로 보여요.

확인된 내용
- 내일 오전 8:40 병원 도착
- 오늘 밤 10시부터 금식
- 신분증 지참

아직 비어 있는 내용
- 혈압약을 먹어도 되는지
- 누가 함께 갈지

가족이 같이 정하면 좋은 것
- 병원에 문의할 사람
- 내일 아침 알림 받을 사람

선택
[내가 확인할게요] [가족에게 물어보기] [알림만 받아두기] [완료로 표시]
```

### 8.2 여정 B: 부모님이 자녀에게 부탁을 명확히 전달

1. 아빠가 보험 문자를 사이에 보낸다.
2. 사이가 "보험료 납부 안내"라고 해석한다.
3. 자동이체 여부 등 원문에 없는 부분을 `확인 필요`로 둔다.
4. 자녀에게 보낼 부탁 문장을 만든다.

출력 예:

```text
자녀에게 부탁할 내용
"보험사에서 보험료 납부 안내 문자가 왔는데,
자동이체가 된 건지 확인이 필요해 보여.
시간 될 때 이 문자 한번 봐줄래?"

첨부 요약
- 보험료 납부 안내
- 금액: 82,000원
- 납부일: 7월 10일
- 확인 필요: 자동이체 여부
```

### 8.3 여정 C: 가족 단톡방에서 역할을 나눔

1. 할머니 병원 예약 안내가 가족방에 올라온다.
2. 사이가 안내를 행동 카드로 바꾼다.
3. 가족이 각자 버튼을 눌러 맡을 일을 정한다.
4. 남은 빈칸만 계속 보인다.

출력 예:

```text
이번 안내에서 해야 할 일
- 예약일 확인: 7월 12일 오전 9시
- 보호자 동행 필요
- 검사 전 금식 필요

가족이 같이 정하면 좋은 것
- 누가 병원에 동행할지
- 검사 후 이동은 어떻게 할지

현재 상태
- 엄마: 할머니께 금식 안내 예정
- 민수: 병원 동행 가능 여부 확인 중
- 지연: 이동 방법 확인 가능

아직 남은 것
- 병원 동행 확정
```

## 9. MVP 범위

처음부터 모든 안내를 처리하려고 하지 않는다. 카카오 대회 출품용 MVP는 다음 4개 문서 유형을 우선한다.

1. 병원/검진 안내
2. 관공서/복지/서류 안내
3. 보험/카드/납부 안내
4. 택배/스미싱 의심 문자

### 9.1 MVP 기능

- 이미지/텍스트 입력
- OCR 또는 비전 기반 텍스트 추출
- 문서 유형 자동 분류
- 행동 카드 생성
- 확인된 사실과 확인 필요 항목 분리
- 가족에게 전달할 짧은 사실 메시지 생성
- 가족 중 누가 확인할지 선택
- 알림 생성
- 완료 체크

### 9.2 MVP에서 하지 않을 것

- 의료 판단
- 법률 판단
- 금융 투자/보험 상품 추천
- 가족 감정 대체
- 자동으로 가족에게 메시지 보내기
- 사용자의 허락 없는 알림 등록

## 10. 기능 설계

### 10.1 입력

입력 방식:

- 문자 붙여넣기
- 카카오톡 메시지 전달
- 이미지/캡처 업로드
- 사진 촬영
- 음성으로 "이거 뭐야?" 질문

이미지 입력은 필수에 가깝다. 실제 사용 상황에서 부모님은 안내 문자를 복사하기보다 캡처를 보낼 가능성이 높다.

### 10.2 문서 유형 분류

문서 유형:

- 병원/검진
- 관공서/복지/서류
- 보험/카드/납부
- 택배/배송
- 스미싱 의심
- 아파트/관리사무소
- 기타 안내

MVP에서는 상위 4개만 깊게 처리한다.

### 10.3 행동 카드 스키마

공통 카드:

```text
무슨 내용?
확인된 내용
해야 할 일
언제까지?
준비물/돈
위험하거나 애매한 점
가족이 도와줄 일
보낼 말
알림
상태
```

### 10.4 문서 유형별 체크 포인트

병원/검진:

- 예약일
- 도착 시간
- 금식 시작 시간
- 물/커피/약 복용 여부
- 신분증/서류
- 보호자 동행 여부
- 검사 전후 주의사항

관공서/복지/서류:

- 신청 대상
- 마감일
- 필요 서류
- 제출 방법
- 방문 장소
- 온라인 신청 링크
- 놓치면 생기는 일

보험/카드/납부:

- 금액
- 납부일
- 자동이체 여부
- 미납/연체 여부
- 고객센터 확인 필요 여부
- 개인정보/인증 요구 여부

택배/스미싱:

- 운송장 번호
- 배송 상태
- 링크 클릭 요구
- 개인정보 입력 요구
- 결제/통관/벌금/긴급 압박 여부
- 공식 앱/사이트에서 확인하라는 안내

### 10.5 확인 필요 분리

사이는 원문에 없는 내용을 단정하지 않는다.

예:

```text
확인 필요
- 혈압약을 검사 당일 아침에 먹어도 되는지는 안내문에 없습니다.
- 병원에 직접 확인하는 것이 필요해요.
```

### 10.6 알림

알림은 단순 리마인드가 아니라 상태 변경을 유도한다.

예:

```text
오늘 21:50
금식 시작 10분 전입니다.
[완료] [30분 뒤 다시 알림]

내일 07:30
신분증 챙길 시간입니다.
[완료] [가족에게 확인 요청]

오늘 18:00
혈압약 복용 여부 확인이 아직 비어 있습니다.
[병원에 물어볼 문장 보기] [가족에게 맡기기]
```

## 11. UI 원칙

### 11.1 어르신용 UI

어르신용 화면은 버튼이 적고 커야 한다.

필수 버튼:

- 사진 찍기
- 이거 뭐야?
- 가족에게 물어보기
- 내가 했어요
- 나중에 알려줘

결과는 길면 안 된다.

```text
오늘 할 일
내일 할 일
가족에게 물어볼 일
```

음성 입력/음성 읽기는 강력한 차별점이 될 수 있다.

### 11.2 가족용 UI

가족용 화면은 조금 더 구조적이어도 된다.

- 원문 보기
- 확인된 내용
- 확인 필요
- 내가 확인할게요
- 가족에게 물어보기
- 알림 만들기
- 완료 체크
- 가족방 공유

### 11.3 버튼 문구

명령형, 감시형, 사무형을 피한다.

좋은 버튼:

- 내가 확인할게요
- 가족에게 물어보기
- 알림만 받아두기
- 완료로 표시
- 짧게 설명하기

피할 버튼:

- 담당자 지정
- 미처리자 독촉
- 부모님 관리
- 효도 메시지 생성

## 12. 안전 원칙

### 12.1 의료

사이는 의료 조언을 하지 않는다. 안내문에 있는 사실을 정리하고, 원문에 없는 것은 확인 질문으로 바꾼다.

예:

```text
혈압약 복용 여부는 안내문에 없습니다.
병원에 확인해야 합니다.
```

### 12.2 법률/행정

법적 판단을 하지 않는다. 신청 대상, 마감, 서류, 문의처를 정리하고, 불확실한 부분은 기관 확인으로 둔다.

### 12.3 금융/보험

상품 추천이나 납부 판단을 하지 않는다. 금액, 납부일, 자동이체 여부, 고객센터 확인 필요를 정리한다.

### 12.4 스미싱

스미싱 여부를 100% 단정하지 않는다.

좋은 표현:

```text
스미싱 위험 신호가 있습니다.
- 링크 클릭 요구
- 개인정보 입력 요구
- 긴급한 결제 압박
공식 앱이나 고객센터로 직접 확인하세요.
```

## 13. 왜 따라 하기 어렵게 만들 수 있는가

OCR 요약은 누구나 만들 수 있다. 사이의 방어력은 다음 자산에서 나온다.

1. **생활문서 타입별 체크 스키마**  
   문서 유형마다 놓치면 손해 보는 필드가 다르다.

2. **확정/추정/확인 필요 분리**  
   원문에 있는 것과 없는 것을 분리한다.

3. **가족 말투가 아닌 가족 전달 구조**  
   감정 흉내가 아니라 상대가 이해할 수 있는 사실 메시지로 변환한다.

4. **상태와 닫힘 추적**  
   안내문이 읽힘에서 끝나지 않고 완료 상태까지 간다.

5. **가족 역할 기억**  
   병원은 누가 주로 확인하는지, 서류는 누가 보는지, 알림은 누가 받는지 쌓인다.

6. **카카오톡 네이티브 흐름**  
   가족방, 전달, 알림, 버튼, 공유에 맞춘 UX가 중요하다.

## 14. 구현 방향

### 14.1 MCP 서버 역할

MCP 서버는 다음 도구를 제공하면 된다.

- `analyze_notice`: 텍스트/이미지에서 안내 카드 생성
- `classify_notice_type`: 문서 유형 분류
- `extract_action_items`: 해야 할 일과 마감 추출
- `detect_missing_fields`: 확인 필요 항목 추출
- `detect_risk_signals`: 스미싱/개인정보/입금/긴급 압박 신호 추출
- `create_family_message`: 가족에게 전달할 사실 메시지 생성
- `create_reminders`: 알림 후보 생성
- `update_task_status`: 항목 상태 변경
- `get_open_items`: 아직 닫히지 않은 항목 조회

### 14.2 데이터 모델 초안

```ts
type NoticeType =
  | 'hospital'
  | 'government'
  | 'insurance_card_payment'
  | 'delivery_or_smishing'
  | 'apartment'
  | 'other';

type NoticeStatus =
  | 'new'
  | 'needs_confirmation'
  | 'assigned'
  | 'in_progress'
  | 'done'
  | 'blocked'
  | 'not_needed';

interface NoticeCard {
  id: string;
  familyId?: string;
  sourceText: string;
  sourceImageUrl?: string;
  noticeType: NoticeType;
  summary: string;
  confirmedFacts: string[];
  actionItems: ActionItem[];
  missingFields: MissingField[];
  riskSignals: RiskSignal[];
  familyMessageDrafts: FamilyMessageDraft[];
  reminders: ReminderDraft[];
  status: NoticeStatus;
  createdAt: string;
  updatedAt: string;
}

interface ActionItem {
  id: string;
  label: string;
  dueAt?: string;
  assigneeLabel?: string;
  status: NoticeStatus;
  evidence?: string;
}

interface MissingField {
  id: string;
  label: string;
  whyItMatters: string;
  suggestedQuestion?: string;
}

interface RiskSignal {
  id: string;
  label: string;
  severity: 'low' | 'medium' | 'high';
  evidence?: string;
  saferNextStep?: string;
}

interface FamilyMessageDraft {
  audience: 'parent' | 'child' | 'family_room' | 'other';
  style: 'short' | 'plain' | 'question';
  text: string;
}

interface ReminderDraft {
  id: string;
  title: string;
  remindAt: string;
  relatedActionItemId?: string;
}
```

### 14.3 LLM 출력 원칙

LLM은 항상 구조화된 JSON을 먼저 만들고, UI는 JSON을 렌더링한다. 자유형 장문 답변을 직접 노출하지 않는다.

필수 검증:

- 모든 action item은 원문 근거 또는 추론 이유를 가져야 한다.
- 원문에 없는 정보는 `missingFields`로 보내야 한다.
- 의료/법률/금융 판단은 `saferNextStep`으로 우회해야 한다.
- 감정 대체 문장은 생성하지 않는다.

## 16. 출품 문구 초안

### 16.1 짧은 소개

**사이**는 가족이 받은 어려운 안내문과 부탁을, 서로 바로 이해하고 함께 완료할 수 있는 행동 카드로 바꿔주는 카카오톡 AI입니다.

### 16.2 긴 소개

부모님이 문자 캡처를 보내며 "이거 뭐야?"라고 물을 때가 있습니다. 병원 안내, 관공서 서류, 보험 납부, 택배 문자처럼 놓치면 손해가 생기는 생활 안내는 많지만, 가족끼리 해석하고 기억하고 다시 확인하는 일은 번거롭습니다.

사이는 안내문을 요약하는 데서 멈추지 않습니다. 확인된 내용, 해야 할 일, 마감, 준비물, 위험 신호, 가족이 확인해야 할 빈칸을 행동 카드로 정리하고, 알림과 완료 체크로 끝까지 닫히게 돕습니다. AI가 가족을 대신해 감정적인 말을 하지 않고, 가족이 직접 챙길 수 있도록 사실과 빈칸을 조용히 정리합니다.

### 16.3 심사 포인트 대응

- 창의성: 가족 사이의 생활문서 통역과 닫힘 추적
- 편의성: 카톡 캡처 전달만으로 행동 카드 생성
- 안정성: 원문 기반, 불확실성 분리, 위험 판단 제한
- 대중성: 부모님, 자녀, 보호자, 가족 단톡방 모두에 자연스러운 사용 장면

## 18. 이 문서를 읽는 다음 세션에게

이 제품은 Argus 원형을 그대로 출품하는 방향이 아니다. Argus의 판단 구조는 내부 엔진으로만 사용한다.

보이는 제품은 `사이`다.

핵심을 잃지 말 것:

- 요약기가 아니라 행동 카드다.
- 비서가 아니라 가족 사이의 확인 레일이다.
- 가족 감정을 AI가 대신하지 않는다.
- 부모님을 수동적 대상으로 만들지 않는다.
- 반드시 "닫힘"까지 추적한다.
- 원문에 없는 내용은 단정하지 않는다.
- 카카오톡 안에서 캡처 전달 -> 카드 -> 확인 -> 알림 -> 완료가 자연스러워야 한다.
